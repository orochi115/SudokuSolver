# P0 阶段策略实现与设计说明

本说明记录了 P0 阶段所实现的 10 个核心策略、架构设计及重构优化。

## 1. 实现了哪些策略与核心思路

我们完整实现了以下 10 个策略，并确保它们在全局优先级、去重及难度的升序排序中完全契合：

- **finned-x-wing** / **finned-swordfish** / **finned-jellyfish**：
  - **核心思路**：我们在 `finned-fish.ts` 中实现了一个通用的、基于 base/cover 模型的鳍鱼/寿司鱼检测器。通过遍历 base houses，挑选出不包含在 cover set 里的“鳍（fin）”候选格。当且仅当所有鳍单元均位于同一个宫（box）内时，我们将处于 cover set 中、不属于 base house、且“能看到该宫内全部鳍格”的覆盖格候选数进行消去。若包含普通鱼则为 Finned，若由于缺少角落而退化则为 Sashimi。
- **nice-loop**：
  - **核心思路**：在 `nice-loop.ts` 中实现。通过调用新增的 `searchNiceLoop` 算法，分别执行以强链或弱链为起点的交替环搜索：
    - **连续环 (Rule 1)**：在完美闭环的每个弱链所共享的单元（house）中，清除所有非环候选格（若为格内弱链则在格中清除其他数）。
    - **不连续环双强 (Rule 2)**：两强链在起点相交，代表此格必为该数字，故在此格中清除所有其他候选数。
    - **不连续环双弱 (Rule 3)**：两弱链在起点相交，代表此格必不为该数字，消去此格的数字。
- **xy-chain**：
  - **核心思路**：在 `xy-chain.ts` 中，我们构建了一个专注于双值格的专用 LinkGraph。我们仅挑选出双值格并生成两个数字对应的节点，其内强链为格内两个候选数的转移，外弱链为同行/列/宫的相同数字连边。对该子图运行 alternating chain 搜索，即可获得最纯粹、最稳健的 XY 链，其两端点必定有其一为真，并消去共同可见格。
- **turbot-fish**：
  - **核心思路**：在 `single-digit-patterns.ts` 中复用单数字 Alternating Chain。我们限制搜索长度为 `maxChainLength: 4`（4 节点，3 链接），当搜索到该长度的多宝鱼强链时，对两端点公共可见格进行消除。
- **hidden-unique-rectangle** / **unique-rectangle-type-3** / **unique-rectangle-type-5** / **unique-rectangle-type-6**：
  - **核心思路**：在 `uniqueness.ts` 的 UR 引擎上进行逐型扩展。
    - **Type 3**：顶层两格多出来的候选数，与所在 Unit 里的其他格共同组合成 K 元显数组（Naked Subset），清除该 Unit 里的多余候选。
    - **Type 5**：两个对角或者三个顶点中多出了特定的多余数，该数必居其一，消去共有可见格。我们额外实现了非常严苛的 mask 检验，确保除了被指定多余数的格之外，其余格不含有其他任何 extraneous 候选数，完美保障了 soundness。
    - **Type 6**：对角顶点含有 extra 候选，而其中一个 UR 候选数在对应的两个行列中形成 X翼（X-Wing），消去对角顶层格。
    - **Hidden UR**：如果其中一格为干净的 `{x,y}`，且对角格 `opp` 所在的行列中，UR 候选数之一 `a` 被完全局限在矩形内部，则该对角格必为 `a`，可消去 `b`。

## 2. 复用了哪些既有 detector 及 overlap/boundaries 的改动

- **复用既有 detector**：
  - `xy-chain` 与 `turbot-fish` 均深度复用了 `x-chain`/`aic` 的 alternating chain 连通图搜索机制，没有另行编写臃肿的独立搜索逻辑。
  - `hidden-unique-rectangle` / `unique-rectangle-type-3/5/6` 完全融入了原有 UR 矩形生成器中，通过重用 `allRectangles` 发生器，进行不同多余候选数指纹的过滤与指代，完美收敛。
- **overlap/boundaries 的改动**：
  - **`overlap.ts`**：已实现的 `turbot-fish`、`xy-chain`、`nice-loop` 以及 `unique-rectangle-type-3/5/6`、`hidden-unique-rectangle` 已正式从 `futureMembers` 移入 `members`。
  - **`boundaries.ts`**：`xy-chain` 与 `nice-loop` 标志位由 reserved 改为正式激活（去除了 `reserved: true`）。

## 3. E2/E3/E6 的处理

- **E2（单数字强链族统一）**：我们将 `single-digit-strong-link` 家族的 `overlap.unified` 设置为了 `true`，标志着多宝鱼家族统一通过链引擎进行发现和表达。
- **E3（UR 引擎收敛重构）**：所有新增的 UR 变体均直接集成至 `uniqueness.ts`，基于相同的 allRectangles 对角矩形与 floor/roof 检测机制，实现了完全共享。
- **E6（chain 引擎归属落实）**：
  - `nice-loop` 负责所有连续/不连续 Nice Loops。
  - 为防止 AIC 探测器不合时宜地返回 loops，我们移除了 `aic.ts` 内部 legacy DFS 中私自返回 placement 环的逻辑（使其遇到自环时继续搜索而不直接返回），使得 Nice Loops 的职责纯粹属于 `nice-loop`。

## 5. 难点与处理方式

1. **同族 Nice 环 shared houses 的消除越界问题**：
   - *难点*：若 Nice 环内包含组节点 (group node)，在清除弱链共享 Unit 上的候选数时，如果我们仅计算 endpoints 首个 Cell 的 shared houses，会导致把一些本不属于整个组共享的 Unit（如某端属于列但另一个不属于该列）纳为消除单元，造成 bad elimination 触发 soundness violation。
   - *解决*：重新设计了 `getCommonHousesForNodes(u, v)`，要求一个 house 必须包含节点 `u` 的**所有 cells** 且包含节点 `v` 的**所有 cells**，才算做真正的弱链共享 Unit。
2. **UR Type 5 的多余候选数严苛过滤**：
   - *难点*：在 Type 5 诊断中，如果不去过滤除去 `d` 之外的其他 extra candidates，会导致候选格在 `d` 假定时并不能坍缩到 deadly pattern `{x,y}`，从而引起错误的 bad-elimination。
   - *解决*：我们加入极为严格的 mask 检验，要求包含 `d` 的格的 mask 必须恰好为 `intersect | maskOf(d)`，不含 `d` 的格必须恰好为 `intersect`。
3. **单元测试在初始盘面因无 pencilmarks 导致 Apply 返回 null**：
   - *难点*：高阶策略需基本解法执行完成、候选数精简后方可触发。在 test 中无法像 Worked Example 一样直接借由 S9B 还原。
   - *解决*：构建了完全 deterministic、极速且纯粹的 hand-crafted minimal state 测试。这避免了测试臃肿度，并完美测试了所有 10 种策略的 `apply` 语义正确性。
