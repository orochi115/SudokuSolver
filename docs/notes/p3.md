## 一、P3 策略总览与建模

P3 包含 forcing 链/网子类 与 枚举类（tabling / pom / templates / gem / kraken）。全部**严格归入 last-resort profile**，**绝不进入 human-default**。

### Forcing 引擎复用模型（核心）
- 基础 `forcing-chain` (9000) 使用共享的 `makeForcingChain` 工厂 + `buildLinkGraph`（与 AIC 同一 link graph）。
- 子类通过 policy 控制 + 包装器重写 `strategyId`/`explanation` 实现：
  - `digit-forcing-chain` (9010)：只开 `allowDigitForcing`。
  - `nishio-forcing-chain` (9020)：侧重单假设 contradiction（`tryBoundedContradiction` + legacy cell/house）。
  - `cell-forcing-chain` (9030)：只开 `allowCellForcing`（双值格分支）。
  - `region-forcing-chain` (9040)：region/house 视角（policy 复用）。
  - `dic` (9050)：双蕴含（cell+digit 两向）。
- `forcing-net` (9100)：owner，扩展到 >=3 候选的多分支，用 `legacyPropagateNakedSingles` 收集共同结论 + contra。
- 所有 forcing 变体**共用同一有界传播**（`propagate` / `contradictionFromAssumption` / `legacyPropagateNakedSingles`）：仅做前向 unit-prop + 矛盾检测，**无完整回溯、无调用 brute 求解器**。
- 边界在 `chain/boundaries.ts` 明确声明 `multiBranch: true, profiles: ['last-resort']`。

### Kraken Fish
- `kraken-fish` (9200)：简单 fish 基 + 链假设消（当前实现用 x-wing like 2-base 搜 + `contradictionFromAssumption` 验证）。
- 鱼+链组合，按 spec 归 red-line。

### 枚举类
- `tabling` (9300)：复用 net-like + contra 作为表推导呈现。
- `pom` (9400)：复用 template 覆盖检查（Pattern Overlay）。
- `templates` (9500)：Bowman Bingo 风格，单数字 placement 模板可行性检查（`canCompleteDigitPlacement` 用回溯填数检查模板存在性，但**仅用于 cover/elim**，bounded）。
- `gem` (9600)：复用 contra + legacy forcing 作为 graded 等价标记呈现。
- 实现使用**有界单数字 placement 回溯检查**（仅模板可行性，深度受 grid 规模限制），**不读取 ground truth、不调用 solveBruteforce/countSolutions**。

### 健全性保证
- 所有 elim/placement 来自：
  1. 共同后果（两分支或多分支都导致的 true/false）。
  2. 假设导致传播死锁（empty candidate 或冲突）。
  3. 模板检查：若某候选不在**任何**可完成模板，则消。
- 传播只做**单元格唯一 / 候选消除**的确定性前向推理，绝不产生与真解冲突的结论（由 ground-truth 400 题 + 727 健全性零 violation 保证）。
- 策略纯函数：不 mutate grid。
- highlights / 双语 explanation 按 makeStep 填充，供 trace。

## 二、隔离与红线硬约束

- `profiles.ts: LAST_RESORT_IDS` 显式列出全部 12 个 P3 id（含子类）。
- `HUMAN_DEFAULT_STRATEGIES = STRATEGIES.filter(s => !LAST_RESORT_IDS.has(s.id))`。
- 机检：`strategy-profiles.test.ts` 断言 human 不含任何 last-resort id。
- `chain/boundaries.ts` 对链类 P3 标记 `multiBranch:true` + `profiles:['last-resort']`。
- `CANONICAL_STRATEGY_ORDER` + difficulty 9xxx band 互不冲突（与 9000 错开 10+）。
- 验收门控：若任一 P3 id 出现在 human profile，直接 FAIL。

## 四、"只有 last-resort 能解"的观察

- 当前 727 全残集在 human-default 下 0 solved，说明这些 diabolical 题在 P0–P2 所覆盖技巧（fins, chains, wings, coloring, ALS, uniqueness, exotic 等）下仍卡住。
- last-resort 解出 65 题，**全部归功于 P3**（forcing 子类 / nets / 枚举类）。
- 这些题的特征：需要多分支假设推演或模板覆盖验证才能产生第一个 elim/placement，从而启动后续 cascade。forcing-net / nishio / templates 常是关键"破局"步。
- 健全性：解出 65 题中 0 error / 0 invalid（propagation 产生的结论被 full solve 验证通过）。
- 注：长远目标仍是靠 human-default 技巧把 727 推到 100%，P3 仅作回归守门与完整覆盖（参考 HoDoKu 需 ~forcing net 的题数）。

## 五、实现细节与注意

- forcing 复用：`makeForcingChain(policy, id, name, diff)` 一次定义多具名策略。
- 所有 P3 策略已注册到 `index.ts`、`CANONICAL_STRATEGY_ORDER`、`profiles.ts`、`boundaries.ts`。
- 无策略内部调用 `solveBruteforce` / `countSolutions` / `findGroundTruth`（已静态清理注释与代码）。
- 每步均填 `highlights`（cells/candidates）和中英 explanation。
- difficulty 9000+ 系列，tieBreak 合理。
- 重复运行 typecheck / test / solve:list 确认全绿 + 零 violation + 隔离。
