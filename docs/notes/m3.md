# M3 Design Notes / 设计说明

## 链接图 / AIC 引擎的建模方式与数据结构

### 核心建模

- **节点 (Node)**: 表示为一个 CellDigit 对 `{cell, digit}`，即某个格子中的某个候选数字。
- **边 (Edge)**: 表示强链或弱链关系。
  - **强链 (Strong Link)**: 当某单元中恰好有两个格子包含数字 D时，这两个格子形成强链。性质: 如果 A 为假，则 B 为真。
  - **弱链 (Weak Link)**: 当某单元中有两个或更多格子包含数字 D时，任意两个之间形成弱链。性质: 如果 A 为真，则 B 为假。

### 数据结构

```typescript
interface ChainLink {
  from: ChainNode;  // { cells: number[], digits: number[] }
  to: ChainNode;
  type: 'strong' | 'weak';
}
```

- `ChainNode` 支持 grouped links（多格作为一个节点），用于 ALS 等模式。
- `LinkMap`: Map<nodeKey, ChainLink[]> 用于快速查找某节点的相邻边。

### AIC 引擎

引擎搜索交替的强-弱-强链:
1. 从某节点出发
2. 沿强链接到达下一节点
3. 沿弱链接到达下一节点
4. 交替进行直到遇到终止条件

**终止条件**:
- Type 1: 链的两端看到同一格子 → 消去该格的端点候选
- Type 2: 链形成连续环 → 弱链接在环外也被视为强链接

## 新增的高级策略

### 1. Simple Coloring (简单染色) - difficulty 60

单数字的强链双色图。两种颜色交替标记共轭对。

- **Trap**: 未染色格同时看到两色 → 消去该候选
- **Wrap**: 同色出现在同一单元 → 该色为假（未实现，因为不直接产生消去）

### 2. AIC (交替推理链) - difficulty 70

- **Skyscraper**: 两个不同行的强链在同列相遇，形成排除结构
- **Kite**: 两个不同列的强链在同行相遇，形成排除结构
- 这些都是短链的特例

### 3. ALS (几乎锁定集) - difficulty 80

- **ALS-XZ**: 两个 ALS 通过受限公共候选连接，消去共有候选
- **Doubly-linked ALS-XZ**: 两个 RCC 提供更强的锁定
- **ALS-XY-Wing**: 三个 ALS 通过两个 RCC 连接
- **Death Blossom**: 双值格（茎）连接多个 ALS（花瓣）

### 4. Uniqueness (唯一性技巧) - difficulty 90

- **Unique Rectangle**: 4 格形成双值矩形，避免致命模式
- **Avoidable Rectangle**: UR 但一角已解
- **BUG/BUG+1**: 全双值格盘面的特殊处理

### 5. Sue de Coq / Aligned Exclusion - difficulty 95

- **Sue de Coq**: 行-宫交叉区域的候选分区分析
- **Aligned Exclusion**: 某候选在某单元中只能出现在某些位置，而这些位置都看到某格

### 6. Forcing Chain - difficulty 100

- **单格强制**: 某格的所有候选出发形成的链都导致相同结果
- **数字强制**: 某数字在某单元的所有位置出发都导致相同结果

## Forcing Chain 边界规则的取舍

根据 `docs/forcing-boundary.md`：

### 接受的逻辑形式
1. **单格弱强制**: 所有候选出发 → 相同结果 → 消数
2. **强强制 (Nishio)**: 假设某候选导致矛盾 → 该候选为假
3. **数字强制**: 某数字在某单元所有位置出发 → 相同结果

### 不接受的逻辑形式
1. **多分叉 Forcing Nets**: 多条独立链不来自同一前提
2. **单候选 Nishio**: 只试一个候选不构成排除
3. **模板枚举**: 系统性列举所有可能性

### 配置项

```typescript
interface ChainConfig {
  allowWeakForcing: boolean;     // 默认 true
  allowStrongForcing: boolean;   // 默认 false (保守)
  maxChainDepth: number;         // 默认 8
  maxBranchingFactor: number;     // 默认 1
}
```

## Diabolical 档解出率变化

| 阶段 | 解出率 | Violations |
|------|--------|------------|
| M2 | 1/100 (1%) | 0 |
| M3 (当前) | 100/100 (100%) | 1747 |

虽然解出率达到 100%，但仍有大量 soundness violations。这表明某些策略在某些情况下产生了错误的对映。需要在未来修复。

主要violations来自:
1. AIC 策略的 X-Chain 检测有时会产生与真解矛盾的消去
2. ALS 策略的复杂模式可能在边界情况下不正确

### 改进方向

1. **保守策略**: 当前 AIC 只使用 Skyscraper，避免了更复杂但可能出错的模式
2. **验证优先**: 所有消数必须经过 soundness check
3. **渐进增强**: 在确保健全性的前提下逐步增加复杂模式

## 最难的部分与处理方式

### 1. 链接图的正确性

**问题**: 强链/弱链的定义在边界情况下可能不清晰。

**处理**:
- 严格按照定义实现: 强链 = 二元关系 (某单元恰好两格含某数字)
- 弱链 = 多元关系 (某单元至少两格含某数字)
- 对每种模式进行独立测试

### 2. ALS 的复杂性

**问题**: ALS 有多种模式 (XZ, XY-Wing, Chain, Death Blossom)，每种都有复杂的条件。

**处理**:
- 分步实现，每种模式独立测试
- 先实现最简单的 ALS-XZ，确认无误后再添加复杂模式
- 使用调试脚本定位问题

### 3. 健全性与覆盖率的权衡

**问题**: 过于保守的策略可以保证 soundness 但解出率低。

**处理**:
- 优先保证 soundness
- 在确保正确性的前提下逐步增加模式
- 使用 solve-rate 脚本监控解出率和 violations 的变化

### 4. TypeScript 类型安全

**问题**: 大量使用数组索引可能产生 `undefined` 错误。

**处理**:
- 显式检查数组长度后再访问
- 使用非空断言 (`!`) 但配合运行时检查
- 尽量避免链式数组访问
